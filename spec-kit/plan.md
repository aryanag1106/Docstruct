# Technical Plan ÔÇö DocStruct

## Non-negotiable principles
1. **CPU-first, always.** No code path requires a GPU/CUDA runtime. `llama-cpp-python`'s default pip build is CPU-only ÔÇö verify this stays true after any dependency bump.
2. **Offline-first, provably.** The pipeline's network-call count is zero during processing; this is a passing automated test (`tests/test_no_network.py`), not just a README claim.
3. **Never fabricate a field.** Low OCR confidence, a missing required field, or a schema validation failure becomes a `needs_review` flag with a reason ÔÇö never a plausible-looking guess.
4. **No real PII, ever.** All sample/test data is synthetic, generated by `scripts/generate_sample_data.py`.
5. **Strong copyleft, deliberately.** GPL-3.0. See `README.md` ┬ž8.
6. **No fake green checkmarks.** Every CI/pre-commit check does real work; a stub (`echo`, bare `exit 0`, a disabled step) is treated as a disqualifying violation, per the hackathon rules.

## Architecture
See `README.md` ┬ž3 for the diagram ÔÇö `ocr.py`/`preprocess.py` Ôćĺ `llm_extract.py` Ôćĺ `validate.py` Ôćĺ `pipeline.py`, with `cli.py` and `app_streamlit.py` as two thin interfaces over the one `pipeline.process_document()` function.

## Why these choices
- **PyMuPDF for text-layer PDFs, Tesseract only when actually needed**: running OCR on a PDF that already has a perfectly good text layer is slower and strictly worse (OCR introduces errors a text layer doesn't have).
- **Tesseract over PaddleOCR for the MVP**: simpler install story (one `apt-get install`), good enough accuracy on printed text, no extra Python ML stack to pin down under time pressure. PaddleOCR remains a documented alternative if accuracy on a particular document type proves insufficient.
- **A generic JSON-grammar (not a fully schema-aware GBNF grammar) for llama.cpp**: guarantees syntactic JSON validity cheaply; semantic correctness (right keys/types) is `validate.py`'s job against the Pydantic schema, and is more reliable to get right there than by hand-writing a schema-to-GBNF compiler under a hackathon clock. Full schema-aware grammar generation is a listed stretch goal, not silently dropped.
- **Qwen2.5-1.5B-Instruct**: small enough to run comfortably on CPU, capable enough to follow a "fill this JSON schema from this noisy text" instruction.

## Stretch goals (explicitly not MVP scope, so nobody scope-creeps into them under time pressure)
- Schema-aware (not generic) GBNF grammar generation per document type.
- Automatic `doc_type` classification instead of user selection.
- Multi-document batch processing in one CLI invocation.
- PaddleOCR as a swappable OCR backend alongside Tesseract.

## Risks & mitigations
| Risk | Mitigation |
|---|---|
| Mock-backend heuristics (regex) look "too easy" / not AI-ish enough | Explicitly labeled as a dev/CI convenience, not the demoed backend; the `llm` backend is what's shown live |
| `llm` backend's actual output quality unknown until a real model is tested against it | `llm_extract()` raises a clear error rather than silently falling back to mock if the model/grammar fails ÔÇö a visible failure during testing is better than a hidden one during the demo |
| Tesseract accuracy on a phone photo (not a flat scan) | `preprocess.py`'s deskew/denoise step exists specifically to close some of this gap; demo uses generated samples close to "good photo" quality, with the limitation declared rather than hidden |
| Two-person team, tight clock | Frozen interfaces (see `docs/work-division.md`) let OCR work and LLM/schema work proceed in parallel from minute one |

## Definition of done
`spec-kit/spec.md`'s acceptance criteria all checked, network disabled for the live demo, on a machine that isn't the one the code was written on.
